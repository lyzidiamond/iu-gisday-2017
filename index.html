Example 2: Mapbox Earthquake Map
  • Data: OpenStreetMap, USGS, served as vector tiles and GeoJSON
  • Styling: designed by cartographers and written in the Mapbox Style Specification
  • Rendering: Mapbox GL JS
Depending on the map we're making and for what purpose, we need to make choices about each of these three aspects.
Data: who collected the data? How is it being stored?
Styling: who decides what the map looks like, and how are those decisions preserved?
Rendering: is the map for print, web, mobile? Will it need to be reproduced on multiple machines?
Each of these aspects can be tweaked to create a map that does what you want it to do.

So let's talk really fast about how the internet works, particluarly in the context of APIs.
API stands for Application Programming Interface.
The idea is: there's a server (computer) listening somewhere on the internet, ready to provide a service or some data. When another computer makes a request to that server, that server knows how to respond.
This is how we can serve map data to multiple devices, making it scalable.
